# prography-11th-backend
prography 11th backend assignment 에 대한 고민했던 흔적/생각, 회고를 정리한 문서입니다.


## 외래키 제약 방식의 사용이유
- 본 프로젝트는 MSA 환경도 아니고 JPA를 사용해서 진행할 것이라 개발 편의성과 데이터의 일관성을 이유로 외래키 제약 방식을 사용했습니다.

## Unique Key 설정 이유
- UK_MEMBER_LOGIN_ID : 로그인은 시스템에서 빈번하게 발생하는 조회임과 동시에 중복 방지 차원에서 유니크 키로 선정
- UK_COHORT_GENERATION : 기수 중복 방지를 위해 선정
- UK_QR_HASH_VALUE : QR코드는 매 기수 출석 시마다 쌓일 경우 데이터가 방대해 질 경우 대비와 중복 방지 차원에서 유니크 키로 선정
- UK_MEMBER_COHORT : 한 기수의 동일 회원이 두번 등록 됨을 방지하기 위함과 조회 성능 최적화를 위해 선정
- UK_SESSION_MEMBER_ATTENDANCE : 세션내의 출결 목록 조회 성능 향상과 중복 출석 방지를 db 차원에서 방지 하기 위해 선정(물론 애플리케이션 레벨에서도 방지)
- 추후 추가적인 인덱스로 성능향상을 노릴 수도 있겠지만, 테이블 성격 및 프로젝트 규모 상 비효율적이라 생각됩니다.

## 멀티모듈 아키텍처 선정 이유 및 패키지 구성 이유
- 멀티 모듈 아키텍처를 활용하여 모듈별 책임에 맞춰 최대한 분할
- module-api : 서비스의 진입점으로 외부 요청 및 응답, 도메인 서비스 조합
- module-domain : 도메인 엔티티 정의 및 도메인 서비스 로직 및 db 연결 및 레포지토리 구성 구성
- module-common : 전역적으로 사용되는 공통 유틸리티, 예외처리, 상수 관리
- infra, boot모듈을 붙이거나 domain을 POJO로 구현하는 등, 여러 초기 아이디어는 방대하였지만 시간 및 규모 관계상 최소화 하였습니다.

## 예외 처리를 도메인 로직에서 진행할지 usecase에서 진행할 지에 대한 고민
- 책임에 맞게 usecase는 비지니스 흐름에서의 예외처리, 도메인 로직은 데이터의 검증 단계에서의 예외처리로 분리하였습니다.

## N+1 문제 방지
- 본 프로젝트는 JPA 연관관계 매핑(다대일)을 사용했기 때문에 usecase 구현 시 최대한 조심스럽게 left/inner join을 써가며 노력해왔습니다.

## 일정별 출결 요약 api 요구사항 의미에 대한 고민
- 파라미터에 sessionId를 받아오고 일정별 회원 출결 요약이라는 네이밍에 한 일정에 여러번 출석체크가 가능하도록 구현하라는 것인가 고민을 많이 하였습니다. 하지만 비지니스 규칙과 "해당 기수(11기) 전체 회원의 출결 통계를 조회합니다."라는 설명에 맞게 sessionId와 무관하게 전체 회원의 출결 통계를 조회하도록 로직을 구성하였습니다.

## AI 사용내용
- 공부하는 의미로 최대한 AI의 힘을 받지 않고 구현하려고 하였으나, 막히는 부분(에러, 실수, 쿼리)에서 도움을 요청하였습니다.
- 전체 API 개발 후 간단하게 코드 검수 작업을 AI를 사용하여 진행하였습니다.
- 시드데이터 init, 통합테스트는 AI로 작성하였고, 검수작업만 진행하였습니다.

## 아쉬운 점
- usecase안에 sequence를 메서드 분리를 해야할까 고민했지만, 번거로움으로 주석으로 대체하였습니다.
- 마감시간에 임박하여 마무리 했기 때문에 swagger나 Docker 빌드 최적화등 진행을 못하였습니다.
- 최대한 처음에 요구사항을 자세히 확인하고 진행하였지만, ERD 및 아키텍처 설계에서 수정해야 될 점이 조금씩 생겨서 아쉬웠습니다.
